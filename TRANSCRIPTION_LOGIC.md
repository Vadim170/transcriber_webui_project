# Логика транскрибирования — детальная документация

> Последнее обновление: март 2026

---

## Обзор архитектуры

Транскрибирование работает **полностью на сервере** (Python). Браузер только запускает/останавливает сессию и опрашивает состояние — он не захватывает и не отправляет аудио.

```
Устройство (mic/remote)
    │  sounddevice InputStream (~30 мс блоки)
    ▼
AudioStreamWorker
    │  ресемплинг → 16 кГц моно int16
    │  WebRTC VAD + RMS fallback (каждые 30 мс)
    ▼
IntervalCutCoordinator
    │  накапливает фреймы в буфер
    │  решает, когда «нарезать» интервал
    ▼
BG-очередь (64 слота)
    ▼
Background transcribe worker
    │  нарезка на 30-секундные куски
    │  вызов ASR-бэкенда
    ▼
intervals.jsonl + combined.jsonl
```

## 1. Захват аудио

| Параметр | Значение | Где задано |
|----------|----------|------------|
| Формат захвата | `int16`, mono (стерео → среднее) | `transcriber.py` — `DTYPE`, `_to_mono_16k_i16` |
| Sample rate устройства | `default_samplerate` (обычно 48000) | определяется `sounddevice` |
| Целевой sample rate | **16 000 Гц** | `TARGET_SAMPLE_RATE` |
| Размер блока (`blocksize`) | `round(device_sr × 30 / 1000)` | ≈ 480 сэмплов при 16 кГц, ≈ 1440 при 48 кГц |
| Latency | `"low"` | `sounddevice.InputStream(latency="low")` |
| Размер очереди | **2048** элементов (один элемент = один callback-блок) | `INTERVAL_DEFAULTS["audio_queue_size"]` |
| Таймаут чтения очереди | **0.2 с** | `_audio_worker` loop |

**Callback → очередь:** каждый `sounddevice`-callback делает `queue.put_nowait((timestamp, data_copy))`. Если очередь полна — инкрементируется счётчик `dropped_chunks`.

**Буферизация на 480 сэмплов:** рабочий цикл накапливает данные в `_partial_buf` и формирует ровно **480 сэмплов** (= 30 мс при 16 кГц) перед передачей в VAD и координатор.

## 2. Детектор голосовой активности (VAD)

VAD используется **не** для выдачи текста, а для определения момента «нарезки» интервала.

### Алгоритм (каждые 30 мс)

```python
# 1. WebRTC VAD
is_speech = vad.is_speech(frame_pcm_bytes, 16000)   # агрессивность = 1

# 2. RMS-фоллбэк: если VAD говорит «тишина», но уровень сигнала ≥ 0.008 → речь
if not is_speech and rms(frame_float) >= 0.008:
    is_speech = True

is_silent = not is_speech
```

| Параметр | Значение |
|----------|----------|
| Библиотека | `webrtcvad` |
| Агрессивность VAD | **1** (шкала 0–3, мягкий режим) |
| Порог RMS-фоллбэка | **0.008** (нормированный float, диапазон −1…+1) |
| Размер фрейма | **30 мс / 480 сэмплов при 16 кГц** |

Зачем фоллбэк: WebRTC VAD может отбросить тихую, но реальную речь. RMS-фоллбэк не позволяет «нарезать» интервал посередине тихого говорения.

## 3. Координатор нарезки интервалов (`IntervalCutCoordinator`)

### Параметры (настраиваются через UI / конфиг)

| Параметр | Умолчание | Смысл |
|----------|-----------|-------|
| `min_interval_s` | **300 с (5 мин)** | До этого момента нарезка по тишине **не происходит** — накапливаем контекст |
| `max_interval_s` | **600 с (10 мин)** | **Принудительная нарезка** по таймеру, независимо от речи/тишины |
| `silence_cut_ms` | **2000 мс (2 с)** | После `min_interval_s`: если все каналы тихие непрерывно ≥ 2 с → нарезать |

### Логика нарезки

```
Старт интервала (t=0)
    │
    ├── [0 … min_interval_s)
    │     VAD-данные накапливаются, тишина НЕ нарезает
    │
    ├── [min_interval_s … max_interval_s)
    │     Если ВСЕ активные каналы тихие ≥ silence_cut_ms
    │         → нарезка в момент (silence_start + silence_cut_ms / 2)
    │           т.е. в середине окна тишины
    │
    └── t = max_interval_s
          → принудительная нарезка прямо сейчас
```

**Многоканальное правило:** тишина считается только когда **оба** активных канала (`mic` и `remote`) одновременно тихие. Если один говорит — отсчёт тишины сбрасывается.

**Минимум для финального флэша:** при остановке, если интервал длился < 0.5 с — он отбрасывается (`flush_current` возвращает `None`).

### Временна́я метка среза

Время среза переводится в **смещение в сэмплах** (при 16 кГц): `sample_offset = cut_time_seconds × 16000`. Буфер разбивается по этому смещению, правая часть остаётся для следующего интервала.

## 4. Фоновая транскрибация

### Путь данных после нарезки

1. **`split_buffer(sample_offset)`** — конкатенирует накопленные 480-сэмпловые фреймы, возвращает левую часть, оставляет правую.
2. Кортеж `(start_dt, end_dt, audio_chunks)` кладётся в **BG-очередь** (maxsize = **64**).
3. Фоновый поток (`_background_transcribe_worker`) забирает с таймаутом **1.0 с** и вызывает `_transcribe_interval`.

### Внутренняя нарезка на 30-секундные куски

```python
_TRANSCRIBE_CHUNK_S = 30.0   # = 480 000 сэмплов при 16 кГц
```

Каждый интервал разбивается на куски по **30 секунд** чтобы не перегружать модель одним длинным запросом. Кусок из нулей пропускается.

### Языковой режим

| Настройка `language` | Поведение |
|----------------------|-----------|
| `"ru"` / `"en"` / … | Один вызов бэкенда с указанным языком |
| `"auto"` (умолчание) | Один вызов в режиме автодетекта |
| `"auto"` + бэкенд поддерживает `supports_multi_candidate` | Три параллельных вызова: `auto` + `ru` + `en`; лучший выбирается по `pick_best_candidate` |

## 5. ASR-бэкенды

Выбор бэкенда определяется строкой модели при старте:

| Бэкенд | Когда используется |
|--------|-------------------|
| **`macos_parakeet`** | model id из `KNOWN_MODELS` (FluidAudio / CoreML на Apple Silicon) |
| **`whisper_cpp`** | путь к `.bin`-файлу или любое другое имя не из `KNOWN_MODELS` |
| `voxtral`, `nemo_asr`, `qwen_asr` | специфичные model id |

**Ключевые параметры Whisper.cpp:**

| Параметр | Значение |
|----------|----------|
| `no_speech_thold` | **0.6** — порог «нет речи» на уровне модели (отдельно от WebRTC VAD) |

## 6. Шина состояния и UI

- **UI опрашивает** `/api/state` каждые **1200 мс** (`setInterval` в `app.js`).
- Захват аудио — полностью серверный, браузер не передаёт аудио.
- Настройки (`min_interval_s`, `max_interval_s`, `silence_cut_ms`, `language`) хранятся в `APP_CFG` и применяются при следующем `/api/start`.

---

## 7. Трекер голосовой активности (`voice_activity_tracker.py`)

Отдельный модуль **не занимается VAD** — он ведёт статистику по уже завершённым интервалам.

- После каждого транскрибированного интервала, если `mic_words > 0` или `remote_words > 0` → вызывается `log_voice_trigger`.
- Данные пишутся в `output_dir/voice_activity.jsonl` (технически один JSON-документ).
- **Ретенция:** события старше **30 дней** удаляются при каждой записи.
- **Статистика:** почасовая (по умолчанию последние **7 дней**), посуточная (по умолчанию последние **30 дней**). Доступна через `/api/voice-activity`.
